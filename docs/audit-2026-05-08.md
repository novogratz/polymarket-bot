# Audit du bot Polymarket smart-money — 2026-05-08

Audit ponctuel à froid sur la branche `main` (commit `ba82556`). Lecture du code réel, pas du CLAUDE.md.
Aucune modification fonctionnelle effectuée — ce document sert de feuille de route pour les correctifs.

## Forces

- **Sizing dynamique cash-floor-aware bien pensé** (`main.py:999-1031`).
  `target_deployed = equity * (1 - cash_floor_pct)` puis `per_slot = remaining_to_deploy / remaining_slots`
  évite la pathologie classique du « premier signal mange tout le cash ». Le quality-multiplier
  est appliqué en aval, pas en amont — propre.
- **Three-pass scan partage la même fenêtre leaderboard+trades** (`main.py:291`, `analyze_smart_money_with_data` x3).
  Pas de facteur 3 sur les appels DataAPI ; strict/relaxed/deep voient exactement la même réalité.
- **Concurrency leaderboard fetch propre** (`smart_money.py:327-337`).
  `ThreadPoolExecutor`, progress logging tous les 50 traders, `_pull` swallow proprement les erreurs
  par-wallet sans bloquer le batch.
- **Anti-pump à l'entrée** (`trading.py:396-407`).
  Compare `entry_price` au `avg_copy_price` du signal — défense importante contre les « queues de pump ».
- **`_sports_event_key` normalisé** (`portfolio.py:278-329`).
  Dédupe robuste via `event_slug` ou regex sur l'URL — évite 6 paris sur le même match.
- **Auto-tuner one-directional** (`auto_tuner.py:49-99`).
  On ne loosen jamais après une série de wins — élimine la pire pathologie ML (sur-fit du bruit).
  Bornes dures (`max(0.04, ...)`, `min(30.0, ...)`).
- **Reverse-lookup tokens à fort flow non-scanned** (`main.py:788-838`).
  Corrige un trou structurel du scan Gamma classique.
- **Lazy-load du SDK clob** (`trading.py:25-41`).
  La CI sans `py-clob-client` peut tourner les tests.
- **Trailing stop + peak-protect + take-profit ladder coexistent sans race** (`main.py:1089-1143`).
  L'ordre des branches (stop_loss → peak_protect → trailing → TP → max_hold) est correctement défensif.

## Risques stratégiques (au-delà du code)

1. **Biais de survie du leaderboard MONTH/WEEK.**
   Les top wallets MONTH ont par construction une variance au-dessus de la moyenne — beaucoup de bons traders + quelques whales chanceux qui ont pris un seul fat tail.
   Le filtre `min_trader_pnl=$1k`/`min_roi=3%` n'élimine pas ce biais : un wallet qui a fait $50k sur un all-in à 1% de proba est encore dans le top.

2. **Pas de calibration prospective des wallets.**
   Le scoring utilise `total_trader_pnl` (passé) — c'est régressif. Pas de Brier score, pas de hit-rate par catégorie, pas de time-decay.
   Tu copies des wallets dont l'edge a peut-être déjà disparu.

3. **Lag structurel sur trades dérivés.**
   Fetch des trades via `/trades?start=now-240min` (`smart_money.py:316`) puis entrée au market.
   `max_chase_premium=13%` accepte de payer 13% au-dessus du fill moyen de la whale.

4. **Exits en cascade dans un crash.**
   Sur un événement type « Trump tweet » qui drop tous les marchés politiques 20%, `stop_loss` + `peak_protect` + `cohort_sold` se déclenchent simultanément sur N positions.
   Pas de rate-limit sur sells par tick, pas de circuit breaker « si > 5 exits en un tick → pause les nouveaux entries ».

5. **Auto-tuner trop optimiste sur 30 trades.**
   Avec n=30, IC à 95% sur la win rate ±18% — le seuil `win_rate < 0.30` peut se déclencher sur du pur bruit.
   La règle `stop_loss > 40% des trades` ignore que les stops sont **par construction** ~30% sur un portfolio avec take-profit ladder agressif.

6. **Anti-corrélation absente.**
   7 positions sur « Trump policy », « Senate GOP », « Fed rate cut » = 1 position macro. Le dédupe `event_slug` ne capture pas.
   Un cycle baissier risk-off peut faire -30% sur les 7 simultanément.

7. **Pas de regime detection.**
   Le bot tourne pareil un dimanche soir sans flow et un mardi avant le CPI.
   `MIN_LIQUIDITY_USD=500` est un floor par-marché, pas un floor systémique.

8. **Hypothèse fragile sur l'API DataAPI.**
   Le code suppose que `/trades?user=X` est exhaustive et fraîche. Aucun fallback, aucune validation.

9. **`noise_fallback` activé est dangereux quand smart-money est down.**
   Voir bug #4. Une panne d'API → trades classés par score Gamma (urgency × tradability), pas par edge.

10. **Pas de tracking de Sharpe / drawdown.**
    Le journal a `realized_pnl` par trade mais aucun **kill-switch en drawdown** (ex : « si equity < 80% HWM → pause »).

## Synthèse

Le bot est solidement architecturé pour le **happy path** (sizing dynamique, 3-pass scan, anti-pump, peak/trailing) mais a des **trous critiques sur les modes dégradés** :

- API DataAPI down → `noise_fallback` génère des trades aléatoires (#4, #6).
- Race ledger entre BTC et smart-money (#2).
- Peak/exit asymétrique qui rogne le PnL en silence (#1).

Et **aucune validation forward-looking de l'edge** :

- Biais de survie du leaderboard.
- Pas de Brier score.
- Pas de drawdown kill-switch.

**Priorités** : QW1-QW10 valent largement les efforts. **MC1 (forward-validation wallets)** est probablement le plus gros levier de PnL à moyen terme.
